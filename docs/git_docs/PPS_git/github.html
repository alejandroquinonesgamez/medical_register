Este documento detalla las estrategias y herramientas fundamentales para mejorar la seguridad de los repositorios en GitHub.

En el desarrollo de software moderno, la filtración accidental de credenciales es un riesgo crítico, por lo que GitHub ofrece soluciones nativas diseñadas para mitigar este peligro mediante la detección y prevención de secretos expuestos. El contenido se estructura en torno a dos mecanismos principales:

  • Secret Scanning
  • Push Protection

A continuación, se exploran la gestión de alertas, los procesos de remediación en caso de filtración y los pasos necesarios para habilitar estas protecciones tanto en entornos personales como corporativos.

1. Github Secrets Scanning

Plataformas como Github (y funcionalidades análogas en GitLab) ofrecen soluciones nativas para mitigar este riesgo. El Secret Scanning es una característica que escanea proactiva y automáticamente los repositorios en busca de secretos expuestos en cada commit.

Trabaja en asociación con múltiples proveedores de servicios en la nube (AWS, Google Cloud, Stripe, OpenAI, etc.). Si detecta un token de un proveedor conocido expuesto públicamente, GitHub notifica directamente al proveedor y este procede a revocar el token automáticamente para prevenir abusos.

Revisa todo el historial del repositorio alertando a los administradores de la fuga para rotar las credenciales cuanto antes.

En repositorios públicos está activado por defecto y es gratuito. GitHub escanea cada push en busca de secretos de más de 200 proveedores (AWS, Google Cloud, Stripe, etc.). Si encuentra uno, contacta al proveedor para que invalide la clave automáticamente.

En repositorios privados no siempre viene activado. Si usas GitHub Enterprise Cloud, los administradores deben activarlo en la configuración de seguridad.

Para usuarios gratuitos o Pro en repositorios privados, GitHub permite activarlo manualmente desde la configuración del repositorio o de la organización.

1.1. Gestión de Alertas

Cuando GitHub detecta un secreto "en reposo" (que ya estaba en el repo), genera una alerta. Tienes tres formas de cerrarla:

Resolved as real secret
Confirmas que era real, lo revocas/cambias y limpias el historial.
Used in tests
Indicas que es un secreto falso usado solo para pruebas.
False positive
Indicas que el patrón detectado no es realmente un secreto.

2. Push protection

Push Protection es una funcionalidad de seguridad que actúa de forma preventiva, bloqueando el código que contiene secretos antes de que llegue a entrar en el historial del repositorio.

A diferencia del escaneo tradicional (que detecta el secreto cuando ya es tarde), Push Protection impide que el error llegue a ocurrir.

Cuando un desarrollador ejecuta git push, GitHub intercepta los datos en sus servidores y realiza un escaneo ultrarrápido contra una base de datos de patrones de alta confianza (tokens de AWS, Azure, Google Cloud, Stripe, etc.).

Escaneo
Se analizan los cambios en los archivos.
Detección
Si se encuentra un secreto que coincide con un patrón conocido.
Bloqueo
GitHub rechaza el envío de código y devuelve un mensaje de error directamente a la terminal del desarrollador.

2.1. Tipos de protección

Push protection tiene dos tipos:

Para Secretos de Proveedores (Partner Patterns)
GitHub colabora con más de 200 proveedores. Estos patrones son de "alta confianza". Si intentas subir una AWSSECRETACCESSKEY, GitHub la reconocerá inmediatamente porque el formato es único y validado por el proveedor.
Para Patrones Personalizados (Custom Patterns)
Disponible en GitHub Enterprise. Permite que una organización cree sus propias reglas (usando expresiones regulares o Regex) para detectar secretos internos, como llaves de bases de datos propias o tokens internos de la empresa.

2.2. Excepciones o bypass

GitHub entiende que a veces el bloqueo puede ser un falso positivo o que el secreto es necesario para un test específico. Cuando el push es bloqueado, la terminal muestra un enlace o instrucciones para saltarse la protección indicando un motivo:

Used in tests
El secreto no es real, es para pruebas.
False positive
La cadena de texto parece un secreto pero no lo es.
Will fix later
El desarrollador asume el riesgo (opción menos recomendada).

Cada vez que alguien hace un bypass, se genera un registro detallado para los administradores de seguridad.

2.3. Alertas y Gestión

Cuando ocurre un bloqueo o un bypass, se activan los siguientes mecanismos:

Mensaje en Terminal
El desarrollador recibe un error [REJECTED] con una descripción del secreto detectado.
Alertas de Seguridad
Si se realiza un bypass, aparece una entrada en la pestaña Security -> Secret scanning del repositorio marcada como "Bypassed".
Notificaciones
Los administradores reciben correos electrónicos informando que alguien ha ignorado una advertencia de seguridad, permitiendo una intervención rápida.

2.4. Beneficios de Push Protection

Prevención Real
Evita que el secreto toque el servidor. No hay necesidad de limpiar el historial de Git después.
Ahorro de Tiempo
Invalidar un token y limpiar el historial con herramientas como BFG puede tomar horas. El bloqueo toma milisegundos.
Cultura de Seguridad
Educa al desarrollador en el momento exacto en que comete el error (feedback inmediato).
Cumplimiento (Compliance)
Ayuda a cumplir normativas como SOC2 o ISO 27001 al demostrar control estricto sobre las credenciales.
